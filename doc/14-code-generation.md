# Code Generation Strategy

## Overview

UmbraVox uses code generation extensively to produce correct-by-construction implementations from formal specifications. This approach reduces hand-coding errors, enables exhaustive test generation, and supports DO-178C TQL-1 tool qualification.

## Generator Architecture

```
Spec Files (FIPS/RFC/FSM)
    |
    v
Code Generator (Template Haskell / standalone)
    |
    +---> Generated Haskell modules (src/UmbraVox/*)
    +---> Generated test harnesses (test/Test/*)
    +---> Generated FFI stubs (src/UmbraVox/FFI/*)
    +---> Generated documentation (auto-docs)
```

## Generator Qualification (TQL-1)

The code generator is itself a DAL A artifact. Requirements:
- MC/DC coverage of generator logic
- Verified: generated output matches spec for all tested inputs
- Input format: structured FIPS/RFC algorithmic descriptions
- Output format: valid, compilable Haskell modules
- Qualification evidence in `test/evidence/tool-qualification/`

## Formal Input Specification Formats

### `.spec` DSL (Cryptographic Algorithms)

Grammar:
```
algorithm <name> {
  params {
    <name>: <type>      -- types: Bytes, UInt32, UInt64, Bool
  }
  constants {
    <name> = <hex>      -- e.g., K0 = 0x428a2f98
  }
  steps {
    <assignment>*       -- sequential execution
  }
}
```

Assignments use a restricted set of operations: `=`, `^` (XOR), `|` (OR), `&` (AND), `>>>` (right rotate), `<<<` (left rotate), `+mod` (modular addition), `*mod` (modular multiplication), `if-then-else` (conditional). No loops; all iteration is unrolled in the spec.

### `.fsm` DSL (Protocol State Machines)

Grammar:
```
machine <name> {
  states {
    <name>*             -- e.g., Init, Established, Error
  }
  events {
    <name>*             -- e.g., Initiate, BundleReceived, Fail
  }
  transitions {
    <state> + <event> [guard] -> <state> / [action]*
    -- e.g., Init + Initiate [hasBundle] -> KeyFetch / sendRequest
  }
}
```

### `.schema` Format (Serialization)

CDDL (RFC 8610) with extensions for fixed-size byte strings:
```
field: bstr .size N     -- fixed-size byte string of N bytes
```

All schema files are valid CDDL and can be checked with standard CDDL tools in addition to the UmbraVox generator.

## Generator Versioning

Generators are versioned using semver. Every generated output file includes a header comment:
```haskell
-- Generated by UmbraVox.<Generator> v<major>.<minor>.<patch> from <spec_file> at <ISO-8601 timestamp>
-- DO NOT EDIT — regenerate from spec
```

All generated files are checked into git. Re-generation from the same spec file with the same generator version produces byte-identical output (deterministic). This is verified by a CI check that regenerates all files and confirms no diff.

## Equivalence Theorem

For each generator G with specification S and reference implementation R:

> For all valid inputs x: `G(S)(x) == R(x)`

This is verified:
- **Empirically**: all NIST/RFC KAT vectors pass through generated code; 10,000+ random inputs produce identical output between generated and reference implementations.
- **Structurally**: the generator produces the same algorithm steps as the spec (step-by-step correspondence auditable in generated source).

## Code Generation Targets

### A. Cryptographic Primitives

| Primitive | Spec Source | Generated Output | Test Generation |
|-----------|-----------|------------------|-----------------|
| SHA-256 | FIPS 180-4 | Round functions, schedule, padding | All NIST KAT vectors |
| SHA-512 | FIPS 180-4 | Same structure, 64-bit words | All NIST KAT vectors |
| AES-256 | FIPS 197 | SubBytes (S-box), ShiftRows, MixColumns, KeyExpansion | All NIST KAT vectors |
| AES-GCM | SP 800-38D | GHASH, GCTR, GCM-AE/AD | NIST GCM test vectors |
| X25519 | RFC 7748 | Montgomery ladder, field ops (mod 2^255-19) | RFC test vectors |
| Ed25519 | RFC 8032 | Point ops, scalar mult, encoding | RFC test vectors |
| ML-KEM-768 | FIPS 203 | NTT, poly arithmetic, encaps/decaps | FIPS 203 KAT vectors |
| ECVRF | RFC 9381 | Proof gen/verify, output derivation | RFC test vectors |
| HMAC | RFC 2104 | Keyed hash construction | RFC test vectors |
| HKDF | RFC 5869 | Extract + expand | RFC test vectors |
| ChaCha20 | RFC 8439 | Quarter round, block function | RFC test vectors |

**Dual output**: Each primitive generates:
1. **Pure Haskell** (independently verified against NIST KATs and formal specifications; serves as a standalone DAL A verification artifact)
2. **C source code** (production, constant-time) — the same generator produces both Haskell and C from the `.spec` file. The C output uses only fixed-time operations (no branches on secret data, no variable-time memory access). Total C output is <2,000 LOC across all primitives.
3. **FFI binding stubs** (Haskell `foreign import ccall` wrappers connecting to the generated C)
4. **Equivalence tests** (pure Haskell output == FFI output for 10,000+ random inputs)

**Verification approach**: Both output paths are independently verified:
- **Pure Haskell path**: Verified against the complete NIST CAVP / RFC KAT vector suites and cross-checked against formal specifications. This path serves as an independent verification artifact for DAL A.
- **FFI (C) path**: Independently verified against the same NIST KAT vector suites, plus constant-time verification (ctgrind/dudect). The FFI path serves as a second independent verification artifact for DAL A.
- **Cross-validation**: The two paths are additionally cross-validated against each other (10,000+ random inputs must produce byte-identical output), but neither path depends solely on cross-validation for its correctness evidence.

### B. Protocol State Machines

| Protocol | States | Events | Generated Code |
|----------|--------|--------|----------------|
| Signal X3DH | Init, KeyFetch, Computing, Established, Error | Initiate, BundleReceived, Computed, Fail | Session initiation FSM |
| Double Ratchet | Idle, Sending, Receiving, DH_Ratcheting | MsgOut, MsgIn, DH_Step | Chain advancement FSM |
| PQXDH | Init, Classical, Hybrid, Established | DH_Complete, PQ_Complete, Combined | Hybrid agreement FSM |
| PQ Ratchet | Active, Refreshing | MsgCount50, RefreshComplete | Re-encapsulation FSM |
| Truncation | Normal, Snapshot, Attesting, Executing, Failed | EpochEnd, QuorumReached, Timeout, Crash | Epoch lifecycle FSM |

Generated artifacts per FSM:
- Exhaustive pattern matching (all state/event combinations)
- Invalid transition rejection (compile-time where possible, runtime where needed)
- Test cases for every valid transition path
- Property: no reachable state violates invariants

### C. Serialization (CBOR)

Schema definition format:
```haskell
messageSchema = Record "ChatMessage"
  [ Field "version"      UInt8
  , Field "msg_type"     UInt8
  , Field "sender_id"    (FixedBytes 32)
  , Field "recipient_id" (FixedBytes 32)
  , Field "timestamp"    UInt48
  , Field "sequence_num" UInt16
  , Field "message_id"   (FixedBytes 8)
  -- ... remaining fields
  ]
```

Generated:
- CBOR encoder with canonical encoding (deterministic for hashing)
- CBOR decoder with strict validation (reject malformed input)
- Round-trip property tests
- Fuzz targets (malformed input → graceful error, never crash)
- Wire format documentation (auto-generated)

### D. Consensus Logic

| Component | Formal Model | Generated Code |
|-----------|-------------|----------------|
| VRF threshold | Probability spec | `isSlotLeader :: StakeRatio -> VRFOutput -> Bool` |
| Fork choice | Decision tree | `compareForks :: Chain -> Chain -> Ordering` |
| Stake calculation | Arithmetic formula | `effectiveStake :: Balance -> Uptime -> Relay -> Punitive -> Stake` |
| Reward distribution | Proportional split | `distributeRewards :: Pool -> [ValidatorScore] -> [Reward]` |
| Fee adjustment | EMA algorithm | `adjustFee :: EMA -> TargetMsgs -> Fee -> Fee` |
| Adaptive controller | Proportional formula | `adjustParams :: CycleMetrics -> AdaptiveParams -> AdaptiveParams` |
| Early truncation | Threshold comparison | `checkEarlyTruncation :: CirculatingSupply -> AdaptiveParams -> Bool` |

### E. Test Infrastructure

| Test Type | Generation Method | Coverage Target |
|-----------|------------------|-----------------|
| Unit tests | NIST/RFC KAT vectors | MC/DC |
| Property tests | QuickCheck generators from type specs | MC/DC |
| Fuzz tests | Grammar-based fuzzing from CBOR schema | MC/DC |
| State machine tests | Exhaustive path enumeration from FSM | MC/DC |
| Integration tests | Scenario generators from threat model | MC/DC |
| Economic simulations | Monte Carlo from reward formula | Statistical |

### Test Generation Details

**NIST KAT vectors**: All vectors from the CAVP test suite for each algorithm:
- SHA-256: 129 vectors (short/long msg, Monte Carlo)
- AES-256: 400 vectors (ECB, CBC, GCM encrypt/decrypt)
- ML-KEM-768: 100 vectors from NIST PQC round 3 submissions

**Property tests**: 1,000 cases per property using QuickCheck with `seed=42` for reproducibility. Properties include round-trip (encode/decode), commutativity where applicable, and domain-specific invariants.

**Edge cases**: 50 cases per primitive covering: zero input, maximum-length input, all-ones, all-zeros, alternating bits (0xAA/0x55), and boundary values specific to each algorithm.
